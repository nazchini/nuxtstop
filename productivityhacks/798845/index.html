Brief Summary Of The Metrics Being Measured

We are interested in the hot, cold, and warm cache speeds because CDNs are essentially a cache layer. If you‚Äôre interested in reading more, see the analysis here: metrics measured in Jetpack vs CloudFront and my CDN testing methodology.

Couple findings on why Cloudinary was so much better than Amazon AWS Cloudfront.

Finding #1: Cloudinary‚Äôs DNS lookup is significantly faster than CloudFront‚Äôs DNS lookup.

This was similar to what I found in Jetpack CDN vs Cloudfront. Cloudinary is able to achieve a significantly faster lookup time because it‚Äôs DNS path is more likely to be cached by the intermediate routers. Cloudinary DNS lookup path is ‚Äúres.cloudinary.com‚Äù which seems to be the shared DNS path between most Cloudinary users, so that DNS lookup path is receiving a lot of traffic. While CloudFront‚Äôs DNS lookup path is a very customized ‚Äúd20zaq59cm4c4j.cloudfront.net‚Äù.

What‚Äôs very interesting is Jetpack CDN‚Äôs DNS lookup times were always nearly as fast as the hot cache‚Äôs DNS lookup time (even the cold cache 0.00223 seconds was so close to hot cache 0.0016 seconds), while Cloudinary‚Äôs DNS lookup times took a small hit (hot cache 0.00175 seconds to warm cache 0.0236 seconds). I think this can be attributed to Jetpack CDN‚Äôs immense popularity which means its DNS lookup path ‚Äúi0.wp.com‚Äù is going to be cached much more frequently than Cloudinary.

Finding #2: Cloudinary had a mind blowingly good image compression algorithm

I‚Äôm not exactly sure Cloudinary got it so good. The same 703 KB image was somehow compressed down to 36 KB. That‚Äôs just really good and explains how Cloudinary was able to get much faster Data Transfer times than CloudFront.

Other interesting findings

I already talked about these in previous essays ( Jetpack CDN vs CloudFront) so I didn‚Äôt want to write more paragraphs for these.

  • Generally, the TCP handshake and SSL handshake times are the same regardless of file size or file type. Most CDNs and servers are going to be equally fast on these metrics, unless they are using a slower CPU to do SSL computations.
  • CloudFront‚Äôs warm cache (a request 30 minute after the most recent) is about 2x to 4x slower than hot cache. I found this a lot when testing Jetpack CDN. CloudFront likes to degrade a file from RAM to disk (NVMe or SSD) if it‚Äôs not seen active requests recently.

CloudFront really lagged horribly here at 2.355 vs 0.395 seconds for Cloudinary.

Regarding TCP handshake: this is just the same for any connection. So there‚Äôs not a big absolute time delta.

Regarding DNS lookup: not the biggest factor for the large overall time delta but it‚Äôs still worth mentioning because on a relative basis, Cloudinary outperformed by over 100% (0.0494 seconds vs 0.124 seconds).
